# v0.1.0-rc2.3 release notes

`libknock` v0.1.0-rc2.3 is a release-candidate hardening tag after rc2. It keeps the SDK boundary unchanged and focuses on release packaging, documentation precision, and conservative validation claims.

## Highlights

- Documents the standard source archive plus companion `with-vendor` archive model for offline review, restricted CI, reproducible local audit, and LLM-assisted integration.
- Keeps normal Go module users on the standard archive without checked-in `vendor/` content.
- Clarifies validation boundaries for Linux firewall backends, UDP passive capture, TCP SYN paths, Windows packet tooling, macOS packet capture, long-running fuzzing, and production performance characterization.
- Reinforces that `auth-only` and `knock-auth-only` do not hide the TCP port; they gate application protocol admission.
- Clarifies that plain `iptables` expiry is process-managed, while `nftables` and `ipset-iptables` are preferred when kernel-managed timeout behavior is required.

## Validation status

Repository automation covers unit tests, selected race tests, vet/build gates, nested modules, example builds, documentation link checks, dry-run firewall command checks, and short fuzz smoke where run by the release gate.

This tag does not claim full target-host validation for system firewall mutation, passive packet capture, Windows WinDivert/Npcap paths, macOS BPF/pcap paths, or production performance. Operators should record those results with `docs/validation-template.md` before making deployment-specific claims.
