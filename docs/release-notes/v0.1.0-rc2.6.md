# v0.1.0-rc2.6 release notes

rc2.6 is a conservative release-candidate cleanup pass. It tightens public API consistency, release metadata, and validation wording without claiming new platform production coverage.

## What changed

- Low-level public `knock` entrypoints now normalize a nil `context.Context` to `context.Background()`, matching the tolerance already used by higher-level auth, netx, gate, and relay APIs.
- Removed an unused firewall command helper; backend detection and diagnostics continue to use the Runner-aware command path.
- Updated nested gRPC example/integration modules and the optional Prometheus adapter module to require `github.com/libknock/libknock v0.1.0-rc2.6` while keeping local `replace` directives for source-tree development.
- Release checks keep formatting limited to project-owned Go files outside `vendor/`, build representative examples, run API surface checks, validate vendored release archives, and give explicit guidance when maintainer-only `dupl` is missing.

## Validation run

Maintainer release validation for this tag should include:

```sh
scripts/check.sh
scripts/release-check.sh
scripts/check-api.sh
scripts/package-release.sh --with-vendor v0.1.0-rc2.6 dist/
```

The release gate includes short fuzz smoke and benchmark smoke only. Longer campaigns remain planned validation:

```sh
FUZZTIME=10m scripts/fuzz-long.sh
BENCHTIME=3s BENCHCOUNT=3 scripts/benchmark.sh
```

## Artifact model

Publish both:

- `libknock-v0.1.0-rc2.6.zip` for ordinary Go module users.
- `libknock-v0.1.0-rc2.6-with-vendor.zip` for offline review, reproducible local audit, LLM-assisted integration, and restricted CI.

The vendored archive must contain `vendor/modules.txt`, `go.work`, and `go.work.sum`, and should pass `go test -mod=vendor ./...` and `go vet -mod=vendor ./...`.

## Known validation boundaries

This release does not claim target-host validation for:

- Linux nftables, ipset-iptables, or iptables fallback rule enforcement and cleanup.
- IPv4/IPv6 firewall matrix coverage, repeated startup cleanup, or abnormal-exit stale-rule recovery.
- UDP passive capture with `drop_udp_knock_port`, including root/CAP_NET_RAW/pcap privileges and firewall DROP interaction.
- TCP SYN / TCP SYN sequence raw-packet paths.
- Windows WinDivert/Npcap paths and macOS BPF/pcap paths.
- Long fuzz campaigns and production performance baselines.

Use `docs/validation-template.md` to record real-host evidence before making deployment-specific production claims.
